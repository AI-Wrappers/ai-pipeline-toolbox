# Project Overview
This project is a flexible, highly decoupled Python framework/wrapper for custom generative AI pipelines (Diffusers, Transformers, etc.). 
The core objective is to separate the pipeline logic (generation) from the orchestration, state management, model downloading, and I/O operations.
The framework must be agnostic to the presentation layer, allowing it to be consumed by CLI, Jupyter Notebooks, or GUI applications.

# Architectural Principles
1. **Loose Coupling & Dependency Injection:** Components (`LoopManager`, `StateManager`, `ModelDownloader`, etc.) must not be hardcoded into the pipeline. They are injected or orchestrated by the `Runner`.
2. **Strict Typing:** Use Python type hints, `Generics`, and `Pydantic` for config and workload validation. 
3. **Abstract Base Classes (ABC):** All pipelines and core components must inherit from defined interfaces.
4. **Single Responsibility:** Each module does exactly one thing (e.g., `ResultSaver` only saves, `StateManager` only tracks progress).

# Component Specifications

## 1. Model Registry & Code Generation

* **Source:** Models are defined in `registry/models.yml`.
* **Generation:** A standalone script (`registry/generator.py`) statically generates `generated_enums.py` containing Python `Enum` classes (e.g., `Checkpoints`, `Vae`, `Clip`).
* **Rule:** The agent must run the generation script automatically if `models.yml` is modified. Pipelines use ONLY these Enums, never hardcoded strings.

## 2. BaseGenerationPipeline (The Contract)

* Must be an `abc.ABC`.
* **Requires Generic Types:** `Pipeline[ConfigType, WorkloadType, ReturnType]`.
* **Attributes:** - `required_models: list[Enum]` (populated from `generated_enums.py`).
* **Methods:** - `setup(models_paths: dict)`: Initializes models into memory.
* `__call__(config, workload) -> ReturnType`: Executes the generation.



## 3. Core Wrapper Components

* **ModelFetcher:** - Parses the pipeline's `required_models` and any dynamically requested models from workloads (e.g., LoRAs).
* Downloads missing weights using an auto-managed `aria2c` background daemon via `aria2p` API.
* Provides authorization headers automatically based on the model's `provider` field (`huggingface`, `civitai`).


* **WorkloadProcessor:**
* Validates and standardizes inputs (images, directories, JSON).
* Uses `Pydantic` to parse complex JSON workloads into strongly typed Python objects expected by the pipeline.


* **LoopManager:**
* Implements queue logic (FIFO/Priority).
* Iterates over the workload agnostic of its internal structure (treats it as `Iterable[WorkloadType]`).


* **StateManager:**
* Persists execution state (e.g., using SQLite or local JSON lines).
* Tracks completed, failed, and pending tasks.
* Allows the `Runner` to resume aborted runs without duplicating work.


* **ResultSaver:**
* Receives the `ReturnType` from the pipeline.
* Dynamically constructs output directory trees based on workload metadata (e.g., `outputs/{date}/{batch_id}/{prompt_hash}.png`).


* **Logger:**
* Centralized structured logging (e.g., `loguru` or native `logging` configured with formatters).



## 4. The Runner (Orchestrator)

* Acts as the Facade.
* **Flow:**
1. Accepts a `BaseGenerationPipeline`, a Workload source, and Configurations.
2. Passes Workload to `WorkloadProcessor`.
3. Queries `StateManager` to filter out already completed tasks.
4. Calls `ModelFetcher` (aggregating static and dynamic models) to ensure dependencies are met.
5. Instantiates the Pipeline with downloaded model paths.
6. Hands the remaining tasks to `LoopManager`.
7. Executes the Pipeline, catches errors, logs them.
8. Passes results to `ResultSaver` and updates `StateManager` on success/failure.



# AI Agent Directives (Execution Rules)

* **Do not write monolithic classes.** If a function exceeds 50 lines, reconsider the abstraction.
* **Data Validation:** Always use `Pydantic` models for `Config` and `Workload` inside pipelines to ensure data integrity before GPU execution.
* **Error Handling:** Catch specific exceptions in the `Runner` (CUDA Out of Memory, network drops) and pass them to the `StateManager` to mark tasks as "failed_retry" or "failed_fatal".
* **Documentation:** Generate docstrings for all public methods explaining the expected input and return types.
